# CommissionFlow Ã— Acumatica Integration
## Product Specification v2.0 - Discovery-Driven Approach

---

## Executive Summary

### The Problem with Our Original Approach

Our initial design assumed a "one-size-fits-all" integration using Acumatica's Contract-Based REST API with hardcoded field mappings. However, after deeper investigation into Acumatica's architecture, we've identified a fundamental challenge:

**Acumatica is a highly configurable ERP platform where every implementation is different.**

Key variability points include:
- **Custom Fields**: Companies add custom fields to invoices, customers, and salespeople
- **User-Defined Fields (UDFs)**: Additional attributes that vary by implementation
- **Module Configuration**: Some companies use Projects, others don't; some use Branches, others don't
- **Salesperson Assignment**: Can be at the invoice header, line level, or through a separate commission module
- **Data Sources**: Invoice data might come from AR Invoices, Sales Orders, or custom Generic Inquiries
- **Field Names**: While the API contract is fixed, the *meaning* of fields can vary based on company configuration

### The New Approach: Discovery-Driven Integration

Instead of prescribing what data to pull and how to map it, we will:

1. **Discover** what's available in the specific Acumatica instance
2. **Let the user choose** their data source (REST API entity or Generic Inquiry)
3. **Dynamically map** Acumatica fields to CommissionFlow fields
4. **Preview data** before committing to a configuration
5. **Support multiple data source strategies** for different Acumatica configurations

### Step 1.3: Schema Discovery

Once a data source is selected, we **dynamically fetch the available fields**.

**For REST API Entity:**
```
GET /entity/Default/{version}/Invoice/$adHocSchema
```
This returns all fields including custom fields.

**For Generic Inquiry:**
```
GET /api/odata/gi/$metadata
```
This returns the schema of all exposed Generic Inquiries.

**For DAC-based OData:**
```
GET /api/odata/dac/$metadata
```
This returns all available DACs and their fields.

## Revised Data Model

### Core Changes from Original Design

1. **Replace hardcoded field enums with flexible JSON mapping**
2. **Store discovered schema for reference**
3. **Support multiple data source types**

```prisma
model AcumaticaIntegration {
  id                      String   @id @default(cuid())
  organizationId          String   @unique
  
  // Connection Settings (unchanged)
  instanceUrl             String
  apiVersion              String
  companyId               String
  encryptedCredentials    String
  status                  IntegrationStatus @default(INACTIVE)
  lastConnectionTest      DateTime?
  connectionErrorMessage  String?
  
  // NEW: Data Source Configuration
  dataSourceType          DataSourceType  // REST_API, GENERIC_INQUIRY, DAC_ODATA
  dataSourceEntity        String          // e.g., "Invoice" or "AR-CommissionInvoices"
  dataSourceEndpoint      String          // Full endpoint URL for queries
  
  // NEW: Discovered Schema (cached)
  discoveredSchema        Json?           // Cache of available fields
  schemaLastUpdated       DateTime?
  
  // NEW: Field Mappings (replaces hardcoded field enums)
  fieldMappings           Json            // See FieldMappingConfig type below
  
  // NEW: Filter Configuration
  filterConfig            Json            // See FilterConfig type below
  
  // Sync Schedule (unchanged)
  syncFrequency           SyncFrequency @default(MANUAL)
  syncTime                String?
  syncDayOfWeek           Int?
  lastSyncAt              DateTime?
  nextScheduledSync       DateTime?
  
  // Customer/Project Settings (simplified)
  autoCreateClients       Boolean  @default(true)
  autoCreateProjects      Boolean  @default(true)
  
  // NEW: Unmapped Salesperson Handling
  unmappedSalespersonAction UnmappedAction @default(SKIP)
  defaultSalespersonUserId  String?
  
  // Timestamps
  createdAt               DateTime @default(now())
  updatedAt               DateTime @updatedAt
  
  // Relations
  salespersonMappings     AcumaticaSalespersonMapping[]
  syncLogs                IntegrationSyncLog[]
  
  @@index([organizationId])

type FieldMappingConfig = {

    idField: string;
    nameField?: string;
  };
  
  // Optional mappings
  project?: {
    sourceField: string;
  };
  description?: {
    sourceField: string;
  };

    sourceField: string;
  };
  
  // Custom field mappings
  customFields?: Array<{
    sourceField: string;
    targetFieldName: string;    // What to store it as in CommissionFlow
  }>;
}

// TypeScript type for filterConfig JSON
type FilterConfig = {
  status: {
    field: string;
    allowedValues: string[];    // e.g., ["Open", "Closed"]
  };
  documentType?: {
    field: string;
    allowedValues: string[];
  };

    field: string;
```

## API Implementation Notes

### Dynamic Query Building

Instead of hardcoded queries, build queries dynamically based on configuration:

```typescript
function buildAcumaticaQuery(integration: AcumaticaIntegration): string {
  const { dataSourceType, dataSourceEntity, fieldMappings, filterConfig } = integration;
  
  if (dataSourceType === 'REST_API') {
    // Build REST API query
    const selectFields = extractSelectFields(fieldMappings);
    const filterClauses = buildFilterClauses(filterConfig);
    
    return `/entity/Default/${integration.apiVersion}/${dataSourceEntity}?` +
      `$select=${selectFields.join(',')}&` +
      `$filter=${filterClauses.join(' and ')}`;
  }
  
  if (dataSourceType === 'GENERIC_INQUIRY') {
    // Build OData query for Generic Inquiry
    const filterClauses = buildODataFilters(filterConfig);
    
    return `/api/odata/gi/${dataSourceEntity}?` +
      `$filter=${filterClauses.join(' and ')}`;
  }
  
  // ... DAC_ODATA implementation
}
```

## Migration Path from Original Design

If any configuration was saved using the original hardcoded approach:

1. Create a migration that converts old enum-based settings to new JSON mappings
2. Map old enum values to their corresponding field names
3. Set dataSourceType to REST_API for existing integrations
4. Run validation to ensure mappings are still valid

## Benefits of New Approach

| Aspect | Original Approach | New Approach |
|--------|-------------------|--------------|
| **Flexibility** | Hardcoded fields only | Any field from any source |
| **Custom Fields** | Not supported | Full support |
| **Generic Inquiries** | Not supported | Full support |
| **User Experience** | Assume fields exist | Show actual available fields |
| **Validation** | Fails at runtime | Preview before commit |
| **Maintenance** | Code changes for new fields | Config changes only |
| **Error Handling** | Cryptic API errors | Clear field-level feedback |

## Appendix: Generic Inquiry Setup Guide (For Acumatica Admins)

If a customer needs maximum flexibility, they can create a Generic Inquiry in Acumatica that exposes exactly the fields needed. Here's a template:

**Recommended Generic Inquiry: "CommissionFlow-Invoices"**

Tables:
- ARInvoice (primary)
- Customer (joined)
- Salesperson (joined)
- Project (optional)

Exposed Fields:
- RefNbr (Invoice Number)
- DocDate (Date)
- Status
- DocType
- CustomerID
- CustomerName
- SalespersonID
- Amount
- DocTotal
- ProjectCD
- BranchID
- Any custom fields needed

Filters (built into inquiry):
- Status IN ('N', 'O', 'C') -- Released statuses
- DocType = 'INV'

This inquiry should be marked "Expose via OData" on the Interface Options tab.
